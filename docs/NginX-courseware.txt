NginX в программе курса

Установка и запуск
Запускаем пример
Для начала необходимо установить nginx. Инструкции вы сможете найти для любой ОС:
Windows,
Linux,
macOS.
Для запуска под macOS и Linux вам необходимо перейти в /etc/nginx/ и создать файл nginx.conf.
Наполним его содержимым:

http {
  include mime.types;

  server {
    listen 8080;
    server_name localhost;

    location / {
            # путь до статики
      root /data/www;
      index index.html;
    }
  }
} 
И запустим:

nginx 
Для Windows необходимо запустить nginx.exe. Теперь по адресу http://localhost:8080/ будет доступна ваша страничка.
Чтобы остановить сервер, необходимо запустить:

nginx -s stop


Настройка статики
Если вам нужно раздавать статику из папки, воспользуйтесь директивой try_files:
server {
  root $root;
  set $root /var/www/src;
  set $my_var https://yandex.ru; # Можно делать переменные

  include listen; # ~~Нужно~~ можно настроить отдельный файл с директивой listen, если у вас, например, SSL на поддомены единый

  charset utf-8;

  proxy_connect_timeout       600; # Если нужны кастомные цифры
  proxy_send_timeout          600;
  proxy_read_timeout          600;
  send_timeout                600;

  server_name my-superhost.com;
    server_name my-other-superhost.com; # Да, можно даже много-много доменов и поддоменов

  location = /favicon.jpg {
    root $root;
  }

  location = /robots.txt {
    root $root;
  }

  location /ping {
    add_header content-type "application/json";
    return 200 ' {"status": "ok", "message": "pong", "timestamp": "$date_gmt", "env": "<my-env>"}';
  }

  location /src {
    root /path/to; # Например, /var/www

    add_header Cache-Control no-cache;
    expires 0;
  }

  location / {
    add_header X-App-Host $host; # Если вам важно знать, какой хост вернулся или для дебага
        add_header Last-Modified $date_gmt;
    add_header Cache-Control 'no-store, no-cache, must-revalidate, proxy-revalidate, max-age=0';
    if_modified_since off;
    expires off;

    try_files $uri /index.html;
    add_header Access-Control-Allow-Origin *; # Если надо разрешить всем
  }
} 
А теперь рассмотрим подробнее и с самого начала. С помощью директивы root задаётся, где сервер будет по умолчанию искать файлы. 
root $root;
set $root /var/www/src; 
Из этого фрагмента видно, что порядок неважен (переменная используется до того, как её объявили, но это ничего не сломало). Главное, что root теперь ссылается на переменную $root. Дальше можно её изменять. 
include listen; 
В данном случае listen — это имя файла, который находится в корне nginx: /etc/nginx/listen. Можете указать любой путь, лишь бы у пользователя, который запускает сервер, был туда доступ:
include /my/custom/path/to/config; 
В конфиге может быть всё, что вам нужно. Делать include можно как внутри сервера, так и снаружи:
include backupstream;

server {
    include listen;
} 
Контент файла listen в данном случае содержит в себе просто директивы для задания портов или сокетов, откуда слушать сервер:
listen [::]:80 backlog=2048 ipv6only=off; 
Запись [::]:80 говорит, что сервер работает не просто на 80-м порту, но и слушает все IPv6 соединения. Если вы не хотите, чтобы сервер работал только по IPv6, в файле listen дополнительно пропишите ipv6only=off.
Если у вас прописано несколько различных A (AAAA) DNS-записей с разных хостов на один сервер, можно указывать их в конфиге и работать с конкретными хостами.
server_name my-superhost.com;
server_name my-other-superhost.com; # Да, можно даже много-много доменов и поддоменов 
Для этого напишите server_name host1 host2 или, как в примере выше, перечислите хосты построчно. 
Файл robots.txt и другую подобную статику (например, favicon разных размеров) можно утащить в отдельные локейшены через nginx:
location = /favicon.jpg {
  root $root;
}

location = /robots.txt {
  root $root;
} 
Или сделать роуты на NodeJS в сервере. Например, этот способ подойдёт, если у вас SSR.
Полезными в приложении будут healthcheck-роуты. Благодаря им можно проверить жизнеспособность сервиса. Иногда в один сервис добавляют множество таких проверочных роутов. Например, /ping на отзывчивость самого сервера, /some-service — проверить какой-то внутренний дополнительный механизм и т. д.
location /ping {
  add_header content-type "application/json";
  return 200 ' {"status": "ok", "message": "pong", "timestamp": "$date_gmt", "env": "<my-env>"}';
} 
В примере выше показано, что возвращать можно не только строку return 200 'success';, но и JSON, если задать корректный заголовок. 
Если у вас ПО или файлы лежат в одном месте, но статика вроде JS- или CSS-файлов достаётся из другой папки, можно это явно указать отдельным локейшеном:
location /src {
    root /path/to; # Например, /var/www
    
    add_header Cache-Control no-cache;
    expires 0;
} 
Если необходимо, отключите кеширование на некоторую статику. Такой же приём нужно использовать при получении индексных файлов. Кешировать index.html нельзя, потому что в таком случае пользователь  будет получать старые файлы, даже если вы уже выкатывали обновление. Он будет работать со старым проектом, пока сам не очистит кеш или пока время жизни кеша на index.html не истечёт.
location / {
  add_header X-App-Host $host; # Если вам важно знать, какой хост вернулся или для дебага
    add_header Last-Modified $date_gmt;
  add_header Cache-Control 'no-store, no-cache, must-revalidate, proxy-revalidate, max-age=0';
  if_modified_since off;
  expires off;

  try_files $uri /index.html;
  add_header Access-Control-Allow-Origin *; # Если надо разрешить всем
} 
Допустим, приложение работает через авторизацию в другом сервисе, и вы знаете, что в куках должен быть ключ user_cookie:
set  $redirect_path  https://some-auth-url;

location / {
  if ($http_cookie !~* "user_cookie") {
    return 302 $redirect_path;
  }

  ...
} 
В nginx есть проверка вхождения строк и if. Если не нашли такого ключа в куке, можете сразу редиректить на нужную страницу (например, авторизации).


Настройка прокси
Настраивать прокси придётся часто, поскольку приложения не всегда делают запросы через CORS напрямую в балансировщик бэкенда. 
Почему придётся делать прокси:
проксировать запросы бэкенда;
проксировать запросы в веб-сервер с SSR (например);
поставить nginx перед серверным приложением (Express, Koa, Python Flask, Golang и т. д.) и проксировать запросы до него же.
Настроить прокси без кеширования сразу до вашего сервиса можно, например, так:
    location / {
    proxy_set_header      Host $host;
    proxy_set_header      X-Real-IP $http_x_real_ip;
    proxy_set_header      X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_pass            http://localhost:8081/;
    proxy_set_header      X-Forwarded-Proto $scheme;
    proxy_set_header      Cache-Control no-cache;

    client_max_body_size  128m;

    add_header            Last-Modified $date_gmt;
    add_header            Cache-Control 'no-store, no-cache, must-revalidate, proxy-revalidate, max-age = 0';

    if_modified_since     off;
    expires               off;
    etag                  off;

    add_header X-App-Env  testing;
  } 
Здесь подразумевается, что на http://localhost:8081/ «крутится» ваше приложение. Неважно: Express с SSR или просто «ручками», Python. Просто абстрактное приложение.
Обратите внимание на следующие директивы:
proxy_set_header — позволяет устанавливать заголовки передаваемые проксируемому серверу
add_header **— позволяет устанавливать заголовки клиенту от сервера
client_max_body_size **— ограничивает максимально допустимый размер тела запроса клиента. Если размер превышен — сервер вернет ошибку. Это важно с точки зрения защиты от атак в отказе обслуживания, когда клиент может слать аномально большие запросы для перегрузки сервера.
Если нужно сделать проксирование до другого сервера, делается также несложно:
location /api/ {
    proxy_pass https://some-host/api-or-other-route/;
    proxy_set_header Host some-host;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header Cache-Control no-cache;
    proxy_http_version 1.1;
    client_max_body_size 50m;
} 
Можно обойтись меньшим числом заголовков:
location /api/ {
    proxy_set_header Host some-host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_pass https://some-host/api-or-other-route/;
} 
Вы можете установить дополнительные заголовки как в одну, так и в другую сторону проксирования.
Иногда нужно сделать проксирование до Web Socket серверов. Способ примерно тот же, но соединение должно остаться открытым. Для работы с API чатов в данном курсе, например, сделано вот так:
location /ws/chats/ {
    proxy_pass http://localhost:5000/api/v2/chats/;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "Upgrade";
    proxy_set_header Host $host;
} 
То есть на 5000-м порту по определённому урлу крутится Web Socket сервер и отвечает пользователям. Также можно заметить, что вы ходите в API по /ws/chats/, но по факту всё это попадёт в http://localhost:5000/api/v2/chats/. То есть если сделать запрос /ws/chats/trash, то к конечному серверу он будет такой: /api/v2/chats/trash.
При проксировании до S3 или CDN хранилищ нужно проверить, что на этих сервисах уже нет сжатия, и оно не конфликтует с тем, что включено у вас (по умолчанию gzip). Может оказаться, что ваш nginx «крашнется» при попытке сжать или распаковать пакет из S3. Вот такой способ может помочь:
location /s3/ {
    proxy_set_header Host storage.yandexcloud.net;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_pass https://storage.yandexcloud.net/path/to/test/S3/;
    gzip off;
} 
Если у вас поднят свой DNS-сервер, из прокси можно обращаться напрямую к нему:
location /api/ {
        resolver 127.0.0.1; # По умолчанию не нужно ставить, он определяется автоматически вашей машиной из настроек
    proxy_pass https://some-host/api-or-other-route/;
        ...
} 
Кроме того, могут понадобиться редиректы. Например, урл устарел, но, поскольку посетители сервиса его активно использовали, было бы хорошо оставить его доступным. Можно перенаправить пользователей на другой нужный вам урл:
location /some-1 {
    return 301 https://$host/some;
} 
Переменная $host автоматически подставит хост, с которого пришёл пользователь. Это удобно использовать не только для разных хостов в рамках одного окружения, но и для разных окружений: testing, stable, unstable и другие.
В зависимости от потребностей можно указать нужный статус ответа. Например: 
301 (Moved Permanently, с англ. «перемещён навсегда») — ресурс был перемещён в новое месторасположение на постоянной основе.
302 (Moved Temporarily, с англ. «перемещён временно») — расположение ресурса временно изменено.
Не стоит забывать, что nginx — это очень старый и мощный инструмент, и что у него огромная документация. Не стоит проходить мимо того как, например, nginx приоритизирует локейшены. В документации нет явного описания приоритета (не было), но из достаточного количества примеров это можно понять.
Вначале nginx всегда посмотрит на location = /url, затем на регулярное выражение, а после — на остальное. То есть у регулярных выражений приоритет выше. Более подробные примеры есть в документации или по ссылкам в конце темы.

Настройка логирования
Логирование (журналирование) — инструмент, который поможет быстро найти, что и у кого пошло не так, в какой момент и с какими данными. Мониторинг позволит выявить, когда произошёл инцидент, а по логам можно найти подробную информацию о нём (или просто проверить данные).
Чтобы настроить запись всех логов в определённый файл, нужно использовать директивы access_log и error_log. Описать их можно даже в главном файле nginx (/etc/nginx/nginx.conf):
access_log /path/to/logs/access.log;
error_log /path/to/logs/error.log; 
Также это можно прописать в отдельный файл в блоке server. Тогда логи этого сервера будут попадать в другое место, отличное от дефолтного.
Что делать, если стандартных логов не хватает или сервис для их сбора (например, Graphana) использует для логов другой формат? Настроить для них пользовательский формат. Nginx позволяет это сделать с помощью log_format:
http {
    log_format compression '$remote_addr - $remote_user [$time_local] '
                           '"$request" $status $body_bytes_sent '
                           '"$http_referer" "$http_user_agent" "$gzip_ratio"';
    
    server {
    gzip on;
    access_log /spool/logs/nginx-access.log compression;
    ...
  }
} 
Пример показывает, что «кастомно» созданный формат (в примере это compression) был привязан к access_log в отдельном сервере. 
Есть много разных переменных для задания формата логов. Подробный список можно найти в документации ngnix.  Выберите те, которые подходят для ваших нужд.
Логирование можно делать по условиям:
map $status $loggable {
    ~^[23]  0;
    default 1;
}

access_log /path/to/access.log combined if=$loggable; 
В этом примере логи будут активированы при проверке статуса.
Также есть возможность включить в логах debug режим. Он позволит посмотреть расширенную информацию и полезен при решении больших (или не очень) проблем.
error_log logs/error.log debug; 

Балансировка
Вы написали свой бэкенд (например, на NodeJS) или фронтенд, а он раздаётся с нескольких серверов. Рассмотрим случай с бэкендом. Схема будет примерно такой:
image
Load balancer представляет собой просто ещё один сервер, который перенаправляет запросы на два других. 
Конфигурационные файлы nginx для серверов 192.168.0.1 и 192.168.0.2 идентичные, потому что при раскладке сервиса на каждую из рабочих машин попадёт один и тот же проект (один и тот же код и конфигурационные файлы). 
Допустим, они выглядят вот так:
server {
  listen [::]:80 backlog=2048 ipv6only=off;

  server_name   my.app.com;

  location / {
    proxy_set_header      Host $host;
    proxy_set_header      X-Real-IP $http_x_real_ip;
    proxy_set_header      X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_pass            http://localhost:8081/; # Здесь развёрнут express с SSR
    proxy_set_header      X-Forwarded-Proto $scheme;
    proxy_set_header      Cache-Control no-cache;

    client_max_body_size  128m;

    add_header            Last-Modified $date_gmt;
    add_header            Cache-Control 'no-store, no-cache, must-revalidate, proxy-revalidate, max-age = 0';

    if_modified_since     off;
    expires               off;
    etag                  off;

    add_header X-App-Env  testing; # Кастомный заголовок, можете убрать совсем
  }
} 
Теперь рассмотрим, как правильно оформить балансировку нагрузки. Балансировщик должен «разбрасывать» запросы по этим двум серверам. Оформить сервера в один абстрактный “frontend” можно с помощью блока upstream:
upstream frontend {
    server 192.168.0.1;
    server 192.168.0.2;
}

server {
    location / {
        proxy_pass http://frontend;
    }
} 
Если где-то приложение развёрнуто на сокете, а не на определённом порту, можно настроить upstream на сокет:
upstream frontend {
        server unix:/tmp/user-name-or-other-name;
} 
По умолчанию в nginx включён алгоритм балансировки Round Robin. То есть запросы отправляются просто по кругу. Если до какого-то из двух серверов балансировщик не может достучаться, все запросы автоматически распределяются между оставшимися серверами. В данном случае на один оставшийся. 
Кроме распределения по кругу, существует множество различных алгоритмов: распределять по расположению, по весам (задаётся граф с весами), даже random и много других.
Если вам нужно будет реализовать дополнительный функционал (например, backup server), всегда можно воспользоваться документацией nginx.


Настройка кеширования и сжатия
Один из способов ускорения проекта — кеширование данных. Его применяют как на бэкенде с запросами в базу или от клиента, так и на кешировании со стороны фронтенда на статику. 
Второй способ — ускорить передачу данных. Это можно сделать с помощью сжатия файлов. Чем лучше он сжат, тем меньше весит, а значит, быстрее «долетит» до клиента.
Но это всего лишь способы оптимизации. Проблема со скоростью проекта может заключаться, например, в том, что форма из двух полей и кнопки весит 5 мегабайт (или даже больше). Здесь уже проблема реализации и стоит задуматься, как уменьшит этот вес. Если, конечно, у вас не Three.js со сложной графикой для формы. 
Кеширование
Вернёмся к оптимизации. Начнём с кеширования. Оно позволяет не делать лишние запросы на CDN или S3 сервера, или на те, где размещён фронтенд. Платформа получает способность  обрабатывать большее число пользователей, при этом нагрузка будет минимальной.
Многие проекты состоят из десятков картинок (тяжёлых и не очень), стилей и скриптов. Заново загружать статику при каждом обновлении страницы неверно с точки зрения как UX, так и проектирования системы. Это увеличит количество запросов и трафика, который будет впустую «гоняться» через сервера. 
В nginx кеширование настраивается, как правило, в рамках location у сервера. Чтобы настроить кеширование, используйте директиву expires:
server {
    # Допустим, делаем для css и js
    location ~* ^.+\.(css|js)$ {
        expires 7d;
    }
} 
Значение expires указано в днях. Их можно заменить, например, на секунды: 
server {
    # Допустим, делаем для css и js
    location ~* ^.+\.(css|js)$ {
        expires 31536000s;
    }
} 
Добавить в кеш недостаточно, нужно ещё оповестить клиента (в частности, браузер), когда перезапрашивать тот или иной файл с таким расширением. Для этого используют заголовок Cache-Control. Также подойдёт заголовок Pragma:
server {
    # Допустим, делаем для css и js
    location ~* ^.+\.(css|js)$ {
        expires 31536000s;
        add_header Pragma public;
        add_header Cache-Control "max-age=31536000, public";
    }
} 
Как результат таких настроек, браузер после первого получения всех этих файлов начнёт отписывать в dev-консоли, что данные берутся из локального кеша:

Сжатие
Для хорошего сжатия нужны хорошие алгоритмы. По умолчанию в nginx сжатие происходит с помощью gzip. Но можно сжимать файлы ещё сильнее, причём потеря времени будет небольшой. 
Настроить сжатие с помощью gzip можно в /etc/nginx/nginx.conf в блоке http:
gzip on;
gzip_static on;
gzip_comp_level 5;
gzip_min_length 1024;
gzip_proxied any;
gzip_types text/plain application/xml application/x-javascript text/javascript text/css text/json; 
Описание всех параметров модуля есть в документации (в том числе на русском). Он поддерживается во всех браузерах:

HTTPS
Отсутствие на сайте SSL (Secure Sockets Layer, с англ. «слой защищённых сокетов») в современном вебе считается плохой практикой. Сайты-визитки ещё можно оставить без HTTPS, но нужно понимать, что SEO-роботы будут хуже к таким сайтам относиться. 
Однако если у вас сервис, который занимается оплатой или работает с пользовательскими и любыми другими данными, оставить его без HTTPS опасно. Злоумышленник легко перехватит пакеты в открытом виде. Да и сервисы оплаты не пропустят вашу платформу для интеграции, пока не «заведёте» HTTPS.
Самое простое — это использовать Let's Encrypt. С помощью этого сервиса можно бесплатно сгенерировать сертификат для домена. Есть несколько «но».
Во-первых, эти сертификаты не криптостойкие. То есть если вы решили подключить такой сертификат для крупного сервиса, который работает с персональными данными или биллингом, — лучше всё-таки купить криптостойкий сертификат. Сделать это можно, например на reg.ru. 
Во-вторых, сертификат работает три месяца. Потом его нужно перевыпустить, и так постоянно. Чтобы не делать это вручную, придётся написать скрипт, который будет сам его обновлять и добавлять в нужную директорию.
Перейдём к практике. Чтобы сервер обрабатывал запросы по защищённому соединению, нужны сертификаты. Допустим, вы их уже купили или сгенерировали, и у вас есть два файла: приватный и публичный ключи. Настроим их в nginx:
server {
    listen [::]:443 ssl backlog=2048 ipv6only=off;

    ...

    ssi on;
  add_header Strict-Transport-Security "max-age=31536000;";
  ssl_ciphers HIGH:!RC4:!aNULL:!eNULL:!MD5:!EXPORT:!EXP:!LOW:!SEED:!CAMELLIA:!IDEA:!PSK:!SRP:!SSLv2;
  ssl_prefer_server_ciphers on;
  ssl_protocols TLSv1 TLSv1.1 TLSv1.2;
  
  ssl_certificate /etc/ssl/cert.crt; # Путь до сертификата
  ssl_certificate_key /etc/ssl/key.key; # Путь до ключа
} 
Главными параметрами будут пути к сертификату и ключу. Всё остальное — это дополнительные параметры для более безопасной настройки или чтобы ускорить загрузку у пользователя, так как SSL — это дополнительный «слой» перед получением информации с проверкой валидности сертификата.
Указание протоколов тоже не менее важно, потому их обычно указывают в  конфигурационных файлах. Подробнее про параметры модуля SSL для nginx можно почитать в документации.
Кроме того, вам помогут статьи о настройке HTTPS-сервера и о том, как добавить SSL в nginx. 
Иногда можно встретить сигнатуру ssl on. Она уже неактуальна и устарела. Использовать её не стоит.
Допустим, у вас WildCard SSL-сертификат, который работает на все поддомены вашего сайта *.domain.com. То есть вы можете использовать один и тот же сертификат для разных поддоменов (важно проверить, на какой уровень максимально он действует, потому что существуют разные вариации WildCard сертификатов). И у вас есть много различных сервисов на разных поддоменах:
https://service1.domain.com — админка;
https://service2.domain.com — ещё один клиентский сервис;
https://service3.domain.com — это вообще какое-то API, которое «смотрит во внешний мир», и к этому API могут обращаться кто угодно и откуда угодно;
и т. д.
И перед каждым сервисом стоит nginx. На серверах лежит всё тот же WildCard сертификат, который умеет работать для этих поддоменов. Неудобно каждый раз писать внутри сервера одни и те же строчки:
server {
    listen ...;

    ...

    ssi on;
  add_header Strict-Transport-Security "max-age=31536000;";
  ssl_ciphers HIGH:!RC4:!aNULL:!eNULL:!MD5:!EXPORT:!EXP:!LOW:!SEED:!CAMELLIA:!IDEA:!PSK:!SRP:!SSLv2;
  ssl_prefer_server_ciphers on;
  ssl_protocols TLSv1 TLSv1.1 TLSv1.2;
  
  ssl_certificate /etc/ssl/cert.crt; # Путь до сертификата
  ssl_certificate_key /etc/ssl/key.key; # Путь до ключа
} 
То есть помимо, например, локейшенов, у вас будет ещё и это тянуться из файла в файл. А если значения поменяются? Менять везде? Неудобно. Потому вынесем это в другой файл. Можно даже вместе с listen:
# /etc/nginx/listen
listen [::]:443 ssl backlog=2048 ipv6only=off;
include locations/500-error; # Допустим, сделали какой-то единый ответ на 500-е запросы для всех сервисов

ssi on;
add_header Strict-Transport-Security "max-age=31536000;";
ssl_ciphers HIGH:!RC4:!aNULL:!eNULL:!MD5:!EXPORT:!EXP:!LOW:!SEED:!CAMELLIA:!IDEA:!PSK:!SRP:!SSLv2;
ssl_prefer_server_ciphers on;
ssl_protocols TLSv1 TLSv1.1 TLSv1.2;

ssl_certificate /var/www/ssl/crt.crt;
ssl_certificate_key /var/www/ssl/key.key; 
Теперь в сервере можно сделать так:
server {
    include listen; # По умолчанию путь берётся из корня: /etc/nginx

    ...
} 
Конфигурационный файл сервера не перегружен лишней информацией. 
Но что делать, если listen нельзя переиспользовать из-за того, что на одном сервере может быть несколько веб-серверов на разных портах, но каждый должен содержать SSL? Вынесем SSL в отдельный файл:
# /etc/nginx/ssl

ssi on;
add_header Strict-Transport-Security "max-age=31536000;";
ssl_ciphers HIGH:!RC4:!aNULL:!eNULL:!MD5:!EXPORT:!EXP:!LOW:!SEED:!CAMELLIA:!IDEA:!PSK:!SRP:!SSLv2;
ssl_prefer_server_ciphers on;
ssl_protocols TLSv1 TLSv1.1 TLSv1.2;

ssl_certificate /var/www/ssl/crt.crt;
ssl_certificate_key /var/www/ssl/key.key; 
Теперь его можно переиспользовать как угодно:
# /etc/nginx/listen

listen [::]:443 ssl http2 backlog=2048 ipv6only=off;
include locations/500-error;
include ssl; 
Или так:
# /etc/nginx/sites-enabled/some-app.static.conf

server {
    listen [::]:5000 ssl backlog=2048 ipv6only=off;
    include ssl;

    ...
} 
80 —> 443
Знаете, что это за цифры? В именовании портов тоже есть соглашения. Никто не обязывает вас держать именно такие порты, но браузеры завязаны на некоторые из них. 
Например, когда вы видите в адресной строке браузера http://my.site, это означает, что ваш сервер работает на :80-м порту: http://my.site:80. Просто браузер опускает :80-й порт. Но если ваш сервер запущен на :3000-м порту, то браузер его отобразит: http://my.site:3000. 
Аналогичная ситуация и с :443 портом. Этот порт ассоциируется с HTTPS-соединением. То есть если в браузерной строке https://my.site, значит, сервер слушает :443 порт: https://my.site:443. Никто не мешает запустить SSL и на другом порту: https://my.site:3000. Описанный выше способ подключения SSL работает для всех доступных портов. 
На Wikipedia есть подробный список общепринятых привязок портов. 
Так зачем может понадобиться редирект с :80-го порта на :443-й порт? Зачем вообще нужен :80, если есть :443? 
Для юзабилити пользователя. Он может ввести в адресной строке my.site.com без названия  протокола (то есть не http://my.site.com или https://my.site.com). В таком случае браузер отправит пользователя сначала на HTTP-версию, ведь ему никто не сказал, что надо перейти на HTTPS. 
Браузер скажет, что не смог достучаться до сайта и вернёт примерное следующее:
image
Пользователь решит, что сайт сломался и уйдёт к конкуренту. Он вряд ли подумает «А вдруг сайт доступен на HTTPS?». Не факт, что он вообще понимает значение этого слова.
Именно потому стоит делать редирект с :80-го порта на :443. Реализовать это можно с помощью 301 или 302 кода ответа и return в nginx:
server {
  server_name my.site.com;
  ssi on;
  listen 80;
  return 301 https://$server_name$request_uri;
} 
При этом у вас уже должен быть настроен сервер на :443 порт и HTTPS. Переменная  $request_uri перенесёт все параметры сайта (GET-параметры или hash-адреса #action). 
Разные провайдеры (например, reg.ru, domain.com, godaddy.com) по-разному присылают сертификаты. Но у каждого из них есть подробные инструкции на сайте, как завести сертификат в nginx. Например, на reg.ru.
SSL-сертификат от Letsencrypt также можно выпустить специальной программой — certbot.
На сайте certbot есть описание того, как выпустить сертификат и сконфигурировать nginx, чтобы он его использовал. Чтобы получить пошаговое руководство, выберите веб-сервер и версию ОС.

HTTP/2
В прошлом модуле вы узнали, что такое HTTP/2, а в этом более детально расскажем, как его настроить.
HTTP/2 обязывает вас использовать SSL сертификаты. Без них подключить HTTP/2 не получится. Если они уже есть, для подключения необходимо сделать следующее:
server {
    listen [::]:443 ***http2*** ssl backlog=2048 ipv6only=off;

    ...

    ssi on;
    add_header Strict-Transport-Security "max-age=31536000;";
    ssl_ciphers HIGH:!RC4:!aNULL:!eNULL:!MD5:!EXPORT:!EXP:!LOW:!SEED:!CAMELLIA:!IDEA:!PSK:!SRP:!SSLv2;
    ssl_prefer_server_ciphers on;
    ssl_protocols TLSv1 TLSv1.1 TLSv1.2;

    ssl_certificate /etc/ssl/cert.crt; # Путь до сертификата
    ssl_certificate_key /etc/ssl/key.key; # Путь до ключа
} 
Не увидели отличий от предыдущего урока? А они есть:
listen [::]:443 ***http2*** ssl backlog=2048 ipv6only=off; 
Всё, что нужно сделать — это добавить параметр для listen. И вот у вас готов HTTP/2. Не верите? Установите сертификаты, поставьте этот параметр и проверьте, что ваш сайт отзывается корректно. Для проверки HTTP/2 используйте, например, сервис keycdn.

Классический способ
Обычным способом можно собрать проект и отправить по SSH на удалённый виртуальный сервер, который был создан в прошлом уроке. 
Для удобства стоит добавить SSH-ключи на удалённый сервер, чтобы каждый раз не вводить пароль. Входить на сервер по паролю небезопасно. Как только ваш сервер начал «торчать» наружу, боты начали перебирать пароли к пользователям на сервере (вы сможете убедиться в этом в логах). Не стоит пренебрегать этим. В этой статье вы найдёте набор простых шагов, как отключить вход по паролю на сервер (шаг четвёртый, если точнее). 
Чтобы переслать файлы с одной машины на другую (неважно, ваша она или это другой сервер), можно воспользоваться утилитой SCP. Здесь можно найти примеры использования этой утилиты. Алгоритм простой. Указываете одну свою или другую машину, откуда копировать файлы или папки, и ровно то же самое для места назначения.

~$ scp file.txt user@remote.host:/some/remote/directory # Копируем файл из локальной машины на сервер в указанную папку
~$ scp -r dir1 user@remote.host:/some/remote/directory/dir2 # Копируется папка из локальной машины
~$ scp user@remote.host1:/directory/file.txt user@remote.host2:/some/directory/ # Можно указывать два разных сервера 
Вы копируете файлы уже собранного проекта на удалённый сервер, поднимаете nginx для раздачи статики. Как запустить nginx, подробно рассказали в прошлой теме. 
Если же у вас SSR-приложение, стоит задуматься про supervisor на сервере, который будет автоматически перезапускать приложение, если оно упало. И всё равно стоит поставить nginx перед вашим приложением, так как nginx в любом случае будет более «вынослив» при работе с подключениями, и его можно настроить сразу как балансировщик нагрузки. 
Однако с серверными приложениями (API, SSR и т. д.) стоит сразу сделать с помощью Docker. 
Docker
Даже если в вашем проекте просто статика (не SSR), стоит задуматься об изоляции окружения с помощью Docker. Ваша задача будет заключаться в том, чтобы обернуть приложение (сборку и запуск) в Docker-образ. 
Если будет только статика, следует добавить nginx как внутри Docker, так и на сам удалённый сервер перед образом со всеми SSL-ключами и прочим.
При сборке вы можете пушить собранный образ во внешний Docker Hub (неважно, публичные или приватные репозитории). После пуша нужно будет по SSH удалённо запустить команду внутри сервера для скачивания нового образа и перезапуска контейнера с новым образом. Не забывайте периодически удалять старые образы (например, запустите процедуру по cron -таске).
Такой способ позволяет развернуть ваше приложение на абсолютно любом сервере (Ubuntu, CentOS, Windows) и не заботиться о том, какой пакетный менеджер (например, apt или yum) у той или иной ОС вашего публичного сервера.
Разворачиваем проект частями
Если вы хотите сделать highload-приложение, которое будет доступно всем пользователям, да ещё и настроить систему контроля версий вашего продукта, стоит присмотреться к S3. Про преимущества подхода говорили ранее. Напомним, что S3 позволит не только разделить логику получения статики, но и сделать так, чтобы при обновлении приложения пользователи не получали белый экран из-за того, что на одной машине (образе) старая версия, а на другой — новая. S3 всегда отдаст нужную версию.

Кроме него есть ряд других инструментов, которые также помогут залить папки и файлы на удалённый S3:

CI / CD
Внимание, вопрос:
Кто будет постоянно заниматься выкладкой новой версии проекта на тестовый и «продовый» сервера?
Вы? Ваш коллега? Наймёте руки, чтобы гоняли ваш проект по SCP? Всё это неправильно. Правильно — настроить один раз CI/CD и деплоиться автоматически по различным триггерам (пуш в ветку tst или stable), запускать проверку тестов на каждый пул-реквест.
CI/CD
CI/CD это комбинация непрерывной интеграции (CI) и непрерывного развёртывания (CD) программного обеспечения в процессе разработки. CI/CD позволяет вам сосредоточиться на реализации вашего проекта, не думая о том, что вам нужно вручную прогнать тесты, «выкатить» новую версию на сервер. Кроме того, CI/CD позволяет свести к минимуму человеческий фактор, который может навредить продукту.
Непрерывная интеграция (CI)
Задача CI заключается в том, чтобы оперативно выявить потенциальные баги и ошибки в сборке вашего проекта. CI собирает ваш проект, прогоняет тесты, а также делает всё, что вы хотели бы автоматизировать на какое-либо событие. Как правило, эти события являются действиями внутри системы контроля версий. Новый pull request? Давайте попробуем собрать код в production-режиме и прогоним тесты ещё до того, как он уйдёт на review или в master ветку. Пример простой настройки CI можно посмотреть в статье на Хабре «Continuous Integration для новичков».
Непрерывное развёртывание (CD)
Основная задача непрерывного развёртывания заключается в том, чтобы собрать ваш код и развернуть его на сервере. Пример CD процесса:
Получение кода из системы контроля версий и выполнение сборки.
Сборка кода в production.
Настройка инфраструктуры, установка необходимых пакетов на сервере.
Копирование кода на сервер.
Настройка переменных окружения для сервера.
Развёртывание компонентов приложения (веб-серверы, API-сервисы, базы данных).
Выполнение тестов и откат изменений окружения в случае провала тестов.
Логирование и отправка оповещений о состоянии поставки.
Так что же выбрать для CI/CD?
Вариантов решения проблем много:
настроить GitHub Actions,
написать свою автоматизацию и запустить в Travis CI (org для публичных, com для приватных проектов в git-системах) или других подобных платформах,
переехать в GitLab и настроить CI внутри git-системы.
Каждый из вариантов будет стоить денег. Например, GitHub Actions или Travis CI не будут бесконечно собирать образы и деплоить их бесплатно. Там есть квота, которая быстро расходуется. Квота стоит денег. Аналогично в GitLab CI. 
Потому для коммерческих проектов стоит рассмотреть варианты со своим Jenkins или GitLab, в котором вы настроите всё так, как вам нужно. Здесь вы платите только за сервер для этого ПО и за дополнительное железо (если понадобится, для небольших проектов можно крутить сборки на том же сервере).



Инструкция по деплою в Облако
В этом уроке вы изучите один из простых способов деплоя, с помощью которого сможете собрать приложение и перенести его на Облако.
Что вы используете:
GitHub Actions для сборки проекта.
GitHub Container registry для хранения образов.
Виртуальную машину в Облаке для развёртывания проекта с помощью спецификации Docker Compose.
Настройка репозитория на GitHub
Прежде чем добавлять workflow для сборки проекта, проверьте настройки репозитория.
Убедитесь, что репозиторий публичный
Откройте репозиторий с проектом и перейдите в его настройки (вкладка “Settings”).
Пролистните страницу с настройками вниз до раздела “Danger Zone”.
Убедитесь, что репозиторий публичный (”This repository is currently public”).

Если репозиторий приватный:
Нажмите кнопку “Change visibility”.
Выберите “Change to public” и подтвердите изменения.
Репозиторий должен быть публичным для того, чтобы вы могли скачивать образы проекта из GitHub Container registry.
Проверьте настройки Workflow permissions
Вам нужно убедиться в том, что в настройках репозитория GitHub разрешено создавать packages через GitHub Actions, используя GITHUB_TOKEN. Для этого:
Откройте репозиторий с проектом и перейдите в его настройки (вкладка “Settings”).
В меню слева раскройте раздел “Actions” и перейдите во вкладку “General”.
Пролистните страницу с настройками вниз до раздела ”Workflow permissions”.
Выберите опцию “Read and Write Permissions” и сохраните изменения.

Добавление workflow в GitHub Actions
В вашем репозитории уже используется один workflow, который запускает линтер после открытия пулл-реквеста или пуша в ветки main и dev. Добавьте в этот же каталог (.github/workflows) ещё один workflow и назовите его build_and_push.yaml. 

name: Push to GitHub Container Registry

on:
  push:
    branches:
      - main

jobs:
  build-and-push:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v2
      - name: set lower case owner name
        run: |
          echo "OWNER_LC=${OWNER,,}" >>${GITHUB_ENV}
        env:
          OWNER: '${{ github.repository_owner }}'
      - name: Login to GitHub Container Registry
        uses: docker/login-action@v1
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build and push client image
        uses: docker/build-push-action@v2
        with:
          context: .
          file: Dockerfile.client
          push: true
          tags: ghcr.io/${{ env.OWNER_LC }}/${{ github.event.repository.name }}/client:${{ github.sha }}

      - name: Build and push server image
        uses: docker/build-push-action@v2
        with:
          context: .
          file: Dockerfile.server
          push: true
          tags: ghcr.io/${{ env.OWNER_LC }}/${{ github.event.repository.name }}/server:${{ github.sha }} 
 Этот workflow будет запускаться только после пуша в ветку main. Он собирает проект, используя Dockerfile клиента и сервера, а затем отправляет образы в GitHub Container registry. Для авторизации используется имя вашего профиля на GitHub и GITHUB_TOKEN, который подставляется автоматически. В качестве тегов для образов используется уникальный ID коммита — значение переменной ${{ github.sha }}. Это поможет вам при обновлении кода — каждый раз вы будете получать ссылки на разные образы.
После успешного прохождения этого workflow в списке packages вашего репозитория появятся ссылки на собранные образы для клиента и сервера. Они понадобятся для деплоя приложения на Облаке.
Чтобы получить ссылку на нужный образ:
На главной странице репозитория в меню справа найдите раздел “Packages”.
Выберите образ.
Скопируйте ссылку вида ghcr.io/user-name/repo-name/client(server):<<tag>>.

Создание виртуальной машины и деплой приложения
Чтобы создать виртуальную машину, зайдите в каталог на Облаке, доступ в который вы получили от куратора. Выберите в каталоге сервис Compute Cloud и нажмите кнопку «Создать ВМ».
В интерфейсе создания виртуальной машины:
Введите любое имя и описание виртуальной машины.
В поле «Выбор образа/загрузочного диска» переключитесь во вкладку “Container Solution”.
Нажмите «Настроить».

Перейдите во вкладку “Docker-compose”.
Вставьте спецификацию Docker Compose. Используйте файл docker-compose.yaml из своего проекта. Внесите в него несколько изменений: добавьте ссылки на образы и явно пропишите значения переменных:


    version: "3.9"
    
    services:
        client:
            container_name: prakticum-client
            image: # Добавьте ссылку на образ клиента. Например, ghcr.io/user-name/repo-name/client:03ef5c798c907402f6d109830dd97ec562d99d90
            build:
                context: .
                dockerfile: Dockerfile.client
                args:
                  CLIENT_PORT: 3000
            restart: always
            ports:
                - "3000:80"
            environment:
              - CLIENT_PORT=3000
              - SERVER_PORT=3001
        server:
            container_name: prakticum-server
            image: # Добавьте ссылку на образ сервера. Например, ghcr.io/user-name/repo-name/server::03ef5c798c907402f6d109830dd97ec562d99d90
            build:
                context: .
                dockerfile: Dockerfile.server
                args:
                  SERVER_PORT: 3001
            restart: always
            ports:
                - "3001:3001"
            environment:
              SERVER_PORT: 3001
    
        postgres:
          image: postgres:14     
          ports:
            - "5432:5432"
          environment:
            POSTGRES_PASSWORD: postgres
            POSTGRES_USER: postgres
            POSTGRES_DB: postgres
          volumes:
            - ./tmp/pgdata:/var/lib/postgresql/data
     
После добавления спецификации нажмите кнопку «Применить».
Выставите минимальную конфигурацию: платформа Intel Ice Lake, гарантированная доля vCPU 20%, vCPU — 2, RAM — 2 ГБ.
Укажите имя пользователя и вставьте публичный SSH-ключ. Ключ остальных членов команды вы можете добавить на уже запущенной виртуальной машине.
Нажмите кнопку «Создать ВМ».
После непродолжительного ожидания запустится виртуальная машина, внутри которой поднимутся контейнеры, указанные в спецификации Docker Compose. 
Чтобы убедиться, что контейнеры работают:
Скопируйте адрес виртуальной машины из панели Облака.
Подключитесь к виртуальной машине.
Выполните команду sudo docker ps. В списке запущенных должны отобразиться контейнеры вашего проекта.
Проверить работу приложения можно и без подключения к виртуальной машине. Для этого достаточно перейти в браузере по адресу виртуальной машины с нужным портом. Например, клиент будет доступен по адресу IP-машины:3000, а сервер — по адресу IP-машины:3001. Если вы используете другие порты, укажите их после IP-адреса виртуальной машины.
Что делать, если код обновился
В этом варианте развёртывания веб-приложения не используется автоматический деплой. Но донести изменения на сервер просто — достаточно поменять в спецификации Docker Compose ссылки на образы.
Если вы хотите изменить спецификацию, по которой разворачиваются контейнеры с приложением:
Выберите виртуальную машину с приложением в каталоге на Облаке.
Нажмите кнопку «Изменить ВМ».
В разделе «Спецификация docker compose» внесите необходимые изменения. Например, измените ссылки на образы (обязательно использовать другой тег, иначе изменения не подтянутся), замените номера портов.
Нажмите «Сохранить изменения».
Запуск и перезапуск контейнеров занимают некоторое время. Если приложение не отображается в браузере, подключитесь к виртуальной машине и проверьте состояние контейнеров.

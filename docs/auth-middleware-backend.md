# Проверка авторизации на бэкенде (API ЯП)

Задача: **middleware проверяет, авторизован ли пользователь**; **подключить перед защищаемыми ручками**. Итог по смыслу: **решение о доступе принимает сервер** (cookie → проверка у Практикума), а не «флаг в Redux» на клиенте. Клиентский [`withAuthGuard`](../packages/client/src/hoc/withAuthGuard.tsx) остаётся слоем UX (редирект, скрытие экранов), но **не заменяет** проверку на Node.

---

## Что уже есть:

| Компонент | Назначение |
|-----------|------------|
| [`requirePraktikumAuth`](../packages/server/middleware/requirePraktikumAuth.ts) | Без cookie → **403** `{ "reason": "Forbidden" }`. С cookie → `GET {PRAKTIKUM_API_URL}/auth/user` с заголовком `Cookie`; не **200** или невалидное тело → **403**; сеть/исключение → **502** `{ "reason": "Auth check unreachable" }`. Успех → **`req.praktikumUser`** и `next()`. |
| [`parsePraktikumUser`](../packages/server/middleware/praktikumUser.ts) | Разбор ответа Практикума в `{ id, displayLabel }`. |
| [`types/express-augment.d.ts`](../packages/server/types/express-augment.d.ts) | Тип `Request.praktikumUser`. |
| [`localPraktikumAuthBypass`](../packages/server/middleware/localPraktikumAuthBypass.ts) | Только **не production**: при `LOCAL_PRAKTIKUM_AUTH_BYPASS` подставляет пользователя без запроса к Практикуму (e2e/локальная отладка). В **production** обход **выключен** всегда. |
| [`createApp.ts`](../packages/server/createApp.ts) | Перед **`/api/forum/**`** (через `app.use`), **`GET /friends`**, **`GET /user`** стоит **`requirePraktikumAuth`**. |

Тесты: [`localPraktikumAuthE2E.test.ts`](../packages/server/__tests__/localPraktikumAuthE2E.test.ts) — обход в test/prod, **403** на `/friends` без cookie при выключенном bypass.

## Соответствие формулировке задачи

- **«Middleware проверяет авторизацию»** — выполнено: [`requirePraktikumAuth`](../packages/server/middleware/requirePraktikumAuth.ts).
- **«Перед всеми ручками»** — **частично по букве, по смыслу достаточно для текущего API-сервера:** защищены все **бизнес-ручки**, которые сейчас объявлены в [`createApp.ts`](../packages/server/createApp.ts), кроме **`GET /`** (публичный ответ без данных пользователя). Новые маршруты **нужно** вешать на тот же middleware вручную — иначе они окажутся публичными (см. раздел ниже).

## Важные исправления и пробелы (сделать явно в коде/процессе)

1. **Нет единого «глобального» слоя на всё приложение**  
   Сейчас `requirePraktikumAuth` дублируется на каждом защищённом маршруте/префиксе. Риск: новая ручка в `createApp` **без** middleware останется открытой.  
   **Рекомендация:** либо монтировать все защищённые роуты на один `Router` с `router.use(requirePraktikumAuth)`, либо `app.use` с **явным белым списком** путей без авторизации (`/`, при появлении — `/health` только для пробы без секретов), чтобы «по умолчанию» всё было закрыто.

2. **Публичный `GET /`**  
   Если по ТЗ требуется буквально «каждая ручка», нужно **зафиксировать исключение** в документации/спеке (например: «единственная публичная ручка — liveness») или вынести проверку живости в **`GET /health`** без данных и оставить под политикой команды.

3. **Контракт 502 при недоступности Практикума**  
   Клиенты, ожидающие только **403** для «не залогинен», должны **отдельно** обрабатывать **502** / `Auth check unreachable` (иначе путают с «сессия протухла»). Это уже в спеке форума; при расширении API то же правило стоит повторить в общей HTTP-доке.

4. **Деплой и обход**  
   Убедиться, что в окружениях **production** не выставлены `LOCAL_PRAKTIKUM_AUTH_BYPASS` и родственные переменные (сейчас код их игнорирует, но явный чеклист в CI/CD снижает человеческий фактор).

5. **Документация расхождения 401 vs 403**  
   Сервер для «нет сессии» использует **403** (как в [`forum-api-spec.md`](./forum-api-spec.md) §10). Любые новые интеграции должны следовать этому контракту, чтобы не плодить **401** на тех же сценариях без согласования.

## Желательные улучшения (не блокируют «полноценность» проверки)

1. **Таймаут и наблюдаемость** для `fetch` к Практикуму — сегодня зависание сети может держать запрос долго; разумный `AbortSignal.timeout` + логирование по `requestId` упростят прод-отладку.

2. **Кэширование результата проверки сессии** на короткий TTL (in-memory по cookie или по session id), чтобы снизить число вызовов `GET /auth/user` на бурст запросов — осторожно с инвалидацией при logout.

3. **Отдельный middleware «опциональная авторизация»** (`attachPraktikumUser`: при успехе заполняет `req.praktikumUser`, при неуспехе — `next()` без 403) — для будущих ручек с гостем и автором (см. идею в [`project-themization.md`](./project-themization.md)); для форума и текущих `/friends`, `/user` нужен именно **жёсткий** `requirePraktikumAuth`.

4. **Тесты без bypass** с моком `global.fetch` на успешный/неуспешный ответ Практикума — проверят ветки парсинга и **502**, не полагаясь только на «нет cookie → 403».

5. **CORS + credentials** в dev: браузер шлёт cookie на `packages/server` только при согласованных `credentials: 'include'` и настройках CORS; при смене origin это частая зона поломок — имеет смысл кратко описать в корневом README/env-sample, если команда часто поднимает кросс-портовую схему.

6. **SSR (`packages/client/server`)** не дублирует бизнес-API форума: запросы идут с браузера на API-хост. Убедиться, что в проде cookie-домены и пути согласованы с тем, куда реально ходит клиент (иначе бэкенд «правильный», а cookie не доходят).

## Мини-чеклист перед добавлением новой HTTP-ручки в `packages/server`

- [ ] Ручка действительно должна быть публичной? Если нет — **`requirePraktikumAuth`** (или родительский защищённый router).
- [ ] Нужен ли пользователь в обработчике — только после успешного middleware (`req.praktikumUser` задан).
- [ ] Ошибки безопасности — **403** + JSON `reason` в том же стиле, что форум и существующие ручки.
- [ ] Добавлен ли сценарий в тесты (хотя бы **403** без cookie при выключенном bypass).

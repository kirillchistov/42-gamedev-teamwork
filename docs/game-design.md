# Cosmic Match-3: Дизайн игры

## 1. Концепция
2D-игра в жанре “3 в ряд” (3 in a row / match-3, выбрали по итогам командного голосования, вдохновились [Two Dots](https://www.twodots-game.com/ru/game)).

Игра ведется на фиксированной сетке. 

Игрок обменивает соседние фишки или соединяет цепочки одинакового цвета, чтобы собирать группы и выполнять цели уровня.

В игру могут поиграть на одном устройстве от одного до шестнадцати человек (больше одного необязательно).

## 2. Основные элементы

 - Игровое поле — двумерная сетка (например, 8×8)
 - Фишки разных цветов и типов
 - Комбинации из 3+ одинаковых фишек по горизонтали/вертикали
 - Цели уровня (набрать N очков, собрать M фишек определённого цвета за ограниченное число ходов и т.п.).

## 3. Ключевые сущности

- Поле:  используем дискретную сетку с переменным размером: \`width × height\` (по умолчанию 8×8). 
- Координаты: (col, row) или (x, y), сверху вниз.  
- Типы:  CellPos, GemColor, GemKind, Gem, Cell, Board… дальше видно будет;

## 4. Минимальный gameplay‑цикл (gameLoop)

### 4.1 Онбординг и Ссостояния игры 

- Игрок попадает на экран `/game`.
- Видит краткие правила и кнопку «Старт».
- После старта создаётся экземпляр игры, у игрока появляется поле.
- В процессе игры:
   - Игрок собирает комбинации.
   - Активирует power-ups (полезные и вредные)
   - Набирает очки и меняет позицию в общем рейтинге
- Игра заканчивается при истечении таймера или переполнении поля.
- Показ экрана результата → отправка результата в лидерборд → возможность сыграть снова.

### 4.2 Игровой цикл (упрощённый)
#### Инициализация уровня:
   - Сгенерировать поле без стартовых автоматических матчей (или стартовый каскад).
   - Сбросить счёт, ходы, таймер.
#### **Цикл раунда**:
   1. phase = 'idle': игрок выбирает клетку, затем соседнюю для обмена.
   2. Проверка валидности обмена:
      - Клетки соседние (Манхэттенское расстояние: разница по сумме координат = 1).
      - В результате обмена на поле появится хотя бы одно совпадение 3+.
   3. Если обмен не даёт совпадений:
      - Отменяем (swap back), либо вообще ничего не происходит с фишками.
   4. Если обмен ОК:
      - phase = 'swapping': анимируем перемещение фишек.
      - После анимации - phase = 'resolving', ищем совпадения.
   5. На шаге resolving:
      - Находим все группы совпавших фишек.
      - Рассчитываем очки.
      - Удаляем фишки (помечаем клетки как empty).
   6. phase = 'falling':
      - Для каждого столбца фишки «падают» вниз (сдвиг содержимого к низу).
      - Пустые ячейки сверху заполняются новыми фишками случайных цветов, учитывая возможные ограничения.
   7. Снова phase = 'resolving':
      - Повторяем поиск совпадений.
      - Если матчи есть — повторяем шаги (5–6), пока нет новых совпадений (каскады). \[web:67\]
   8. Когда новых совпадений нет:
      - Проверяем выполнение условий уровня (цели).
      - Уменьшаем movesLeft или обновляем timeLeft.
      - Если цели достигнуты или ходы/время закончились — phase = 'game-over'.
      - Иначе — возвращаемся к phase = 'idle'.

## 5. Алгоритм поиска «3 в ряд»

### 5.1 Простой линейный скан
Простая реализация после каждого хода:

5.1.1 Инициализируем пустую коллекцию совпадений.
5.1.2. Проходим по всем строкам:
   - Идём слева направо.
   - Держим «текущий цвет» и длину последовательности.
   - Как только цвет меняется или строка кончается:
     - Если длина последовательности ≥ 3 — добавляем координаты в matches.
5.1.3. Аналогично проходим по всем колонкам (сверху вниз).

### 5.2 Каскады с постоянным наполнением
Надо описать, что происходит после каждого удаления: удалить/очистить ячейки, включить “гравитацию” (фишки падают вниз), потом добить новыми фишками…

## 6. Управление и ввод
### Минимальный ввод:
- Клик по первой фишке — > выделение.
- Клик по соседней фишке (по стороне) — > попытка обмена.
- Клавиатура: — > стрелками выбирать клетку, Enter или Space — попытка обмена с направлением.

### Логика обмена:
- Проверить, что обе клетки содержат объект (gem).
- Проверить, что разница по координатам == 1 по Манхэттену.
- Сделать временный swap в копии/внутри функции.
- Вызвать findMatches — если матчей нет, отменить swap.

## 8. Интеграция React / Canvas

### 8.1 Единый класс движка Game
- Хранит GameState.
- Реализует:
  - handleClick(screenX, screenY) — преобразует координаты клика в CellPos.
  - trySwap(a: CellPos, b: CellPos).
  - update(dt) — для анимаций (swapping/falling).
  - render(ctx) — отрисовка сетки и фишек ◼, ●, ♦  на канвасе
    - Сетка: линии + подсветка выделенной клетки.
- Вызов колбэков onStateChange и onGameOver.

### 8.2 Компонент GamePage
- Оборачивает Canvas и HUD:
  - Текущее число ходов/времени.
  - Счёт.
  - Цели и прогресс.
- Вешает обработчики:
  - onClick по Canvas, конвертирующий координаты мыши в координаты клетки.

## 9. Этапы наращивания механик
1. **MVP:**
   - Сетка 8×8, 4–5 цветов.
   - Только обычные фишки.
   - Обмен соседних фишек → поиск матчей → удаление → падение → добивка.
   - Счёт + ограничение по ходам, базовый экран Game Over.
2. **Усиления:**
   - Линейные бомбы (4‑в‑ряд).
   - Цветная бомба (5‑в‑ряд).
3. **Фигуры и бомбы:**
   - Детекция квадратов/петель.
   - Превращение внутренних фишек в area-bomb.
   - Взрывы 3×3.
4. **Цели и прогрессия:**
   - Разные конфигурации целей уровней.
   - Блокеры (камни, лёд), клетки‑порталы, и т.п.

Каждый этап (спринт) оформляем отдельным эпиком в бэклоге.

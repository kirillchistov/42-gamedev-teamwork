# Техническое ТЗ по игровым экранам Cosmic Match

Документ описывает целевое поведение игровых экранов в приложении и держится
в актуальном состоянии через ссылки на roadmap:

- `docs/game-improvements-s7.md` — ближайшие продуктовые доработки;
- `docs/game-backlog.md` — статус внедрения;
- `docs/game-design.md` — текущая игровая модель.

## 1. Экранная модель

### 1.1 `/game` — мини-лендинг для авторизованных

Обязательно:

- главная CTA-кнопка "Играть";
- краткие карточки "как играть";
- ссылки на `forum`, `leaderboard`, `profile`.

Цель экрана:

- дать быстрый вход в матч;
- объяснить базовую механику;
- связать игру с остальной частью приложения.

### 1.2 `/game/start` — подготовка матча

Обязательно:

- описание выбранного уровня;
- настройки партии (в рамках доступных опций);
- запуск матча по кнопке.

### 1.3 `/game/play` — активная игра

Обязательно:

- игровое поле Canvas;
- HUD (счет, прогресс цели, ходы/время);
- корректное завершение партии с передачей результата в UI.

### 1.4 `/game/finish` — завершение матча

Обязательно:

- итог матча и причина завершения;
- кнопка "сыграть снова";
- кнопка "в главное меню" (`/game`).

## 2. Интеграция Canvas <-> React

Минимальный контракт:

- инициализация движка в компоненте экрана;
- пуш изменений HUD через callback;
- событие завершения: `onGameEnd({ reason, snapshot })`;
- очистка движка и таймеров при unmount/смене экрана.

## 3. Нефункциональные требования

- состояние матча не ломает роутинг;
- пользователь может безопасно обновить страницу на любом `/game/*` роуте;
- переходы между состояниями не создают дублирующих инстансов движка;
- в UI нет "пустых" тупиковых экранов без следующего действия.

## 4. Улучшения по roadmap (s7+)

Приоритетные доработки:

1. Режим игры по ходам.
2. Расширенные цели и их прозрачный прогресс.
3. Бустеры и post-lose продолжение.
4. Мета-слой и персонаж.

Источник требований: `docs/game-improvements-s7.md`.

## 5. Поддержка актуальности

Чтобы документ не устаревал:

- для каждого пункта ставить статус в `docs/game-backlog.md`;
- при релизе фиксировать изменения в changelog PR;
- обновлять формулировки "Обязательно" только после согласования в команде.

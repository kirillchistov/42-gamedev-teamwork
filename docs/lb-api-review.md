## 2. Рекомендованные улучшения (связь с теми же комментариями)

Это не обязательные перед влитием пункты, а то, что усиливает поддержку и предсказуемость.

1. **Типы API (комментарий №1)**  
   Если бэкенд задокументирует точную схему `data`, можно сузить `unknown` до конкретных типов (числа/строки) на ключевых полях и оставить расширение только для редких полей. Сейчас баланс «гибкость API / безопасность» разумный.

2. **Повторный fetch лидерборда (комментарий №2)**  
   При желании можно добавить отдельный флаг «первая загрузка» vs «тихий рефетч» и показывать полноэкранный скелетон только при первом заходе; для текущего UX достаточно не трогать `data` в `pending`.

3. **Дата на UI (комментарий №4)**  
   В таблице лидерборда сейчас может показываться сырая строка с API. Имеет смысл форматировать для пользователя (`toLocaleDateString` и т.д.) **после** парсинга ISO, чтобы источник истины оставался машиночитаемым.

4. **Ошибка лидерборда (комментарий №6)**  
   Можно добавить кнопку «Повторить запрос» (`dispatch(fetchLeaderboardThunk(...))`), чтобы не оставлять пользователя только с текстом ошибки.

5. **Маппер (связь с №1)**  
   Уже разумно подхватывать `id` из `userId`, если API отдаёт иначе. При появлении новых полей — расширять `LeaderboardApiRowData` и маппер в одном месте.

## 3. Комментарий №7: `useCallback`, `user` и ответ коллеги («всё странно»)

### 3.1. О чём вообще речь (простыми словами)

- **`useCallback`** запоминает **одну и ту же функцию** между перерисовками компонента, пока **не изменятся значения из массива зависимостей**.
- Внутри этой функции используются переменные из окружения, в том числе **`user`**, который приходит из **`useSelector`** — то есть это «текущий пользователь на момент последнего рендера».
- Если **`user` не указать в зависимостях**, React **не обязан** создавать новую функцию при смене пользователя. Тогда `handleGameFinished` может **всё ещё держать внутри себя старый `user`** (например, `null` или прошлый объект) — это называют **устаревшим замыканием (stale closure)**.
- Ревьюер просит либо добавить **`user` в зависимости**, либо по-другому организовать код (например, читать пользователя в момент вызова из стора), чтобы **не отправлять в API устаревшие данные** и не нарушать правило хуков.

### 3.2. Почему замечание ревьюера всё равно справедливо

Даже если по продукту верно: «в игру без логина не попасть» и «в процессе партии нас не разлогинивают», это **не отменяет** механику React:

1. **`user` в Redux может обновиться без «выхода из игры»** — сменили аватар, обновили `display_name`, подтянули профиль с сервера, исправили баг в редьюсере. Тогда колбэк без `user` в deps мог бы **отправить в лидерборд старый ник или старый id**.
2. **Порядок монтирования и асинхронщина** — в редких случаях игра может завершиться в тик, когда стор уже обновился, а колбэк ещё «запомнил» прошлый снимок `user`.
3. **ESLint `react-hooks/exhaustive-deps`** — это не «придирка к авторизации», а **статическая проверка**: «все внешние значения, которые читаете внутри колбэка, должны быть либо в deps, либо явно стабилизированы/получены иначе». Иначе легко получить баг, который на демо не воспроизводится, а в проде — да.

Итого: **`user` в зависимостях — это не про „можно ли разлогиниться в игре“, а про „функция всегда видит актуальный `user` из последнего рендера“`.**

### 3.3. Почему коллега чувствует «странность» — и где он прав

Коллега рассуждает на уровне **бизнес-инвариантов**: если авторизация и маршруты гарантируют сценарий, ему кажется избыточным дублировать это в deps. Это нормальная интуиция: **продуктовая модель** действительно может исключать разлогин в игре.

Но **React не знает** про эти инварианты: он видит только правило «функция замкнулась на `user`». Поэтому:

- **Технический ответ ревьюеру:** да, с `user` в deps поведение корректно и предупреждение линтера снимается.
- **Ответ коллеге:** зависимость `user` не спорит с авторизацией; она лишь говорит: «когда стор с пользователем обновился — пересоздай колбэк, чтобы внутри не остался старый снимок». Это дешёвая страховка.

### 3.4. Альтернативы (если когда-нибудь захочется «меньше deps»)

Все варианты сложнее для новичка; для текущего кода **достаточно оставить `user` в deps**.

- Вынести отправку в **`useEffect`**, который слушает «игра закончилась» + читает `user` из селектора в момент эффекта.
- Внутри колбэка вызывать **`getState()`** из `store` (если есть доступ к стору) и брать `user` оттуда в момент завершения игры — тогда колбэк может не зависеть от `user`, но это уже осознанный паттерн и другие компромиссы.
